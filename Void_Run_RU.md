## Часть 1. Основной документ.

### 1. Заголовок и базовая информация.

• **Название:** Void Run.

• **Назначение:** Проект № 9. Продукт.

• **Фаза проекта:** Phase I.

• **Технологический стек:** Python

• **Статус проекта:** Полностью завершен.

### 2. Краткое описание проекта.

• Проект Void Run – это пародийная версия классической аркады Pacman, разработанная на Python с использованием библиотек Pygame и tcod.

• Игрок управляет желтым героем, который собирает разбросанные по лабиринту "печеньки", увеличивая свой счет. По лабиринту движутся призраки разных цветов, столкновение с которыми отнимает здоровье. При достижении 50 очков открываются ворота, дающие доступ к новой части лабиринта в центре. Цель игры – собрать все 318 печенек и победить.

### 3. Четкие цели проекта.

• Создать пародийную версию классической игры Pacman с уникальным лабиринтом.

• Реализовать полноценную систему перемещения героя с буферизацией ввода и коллизиями со стенами.

• Реализовать движение призраков с использованием алгоритма поиска пути в двумерном массиве A* (библиотека tcod).

• Ввести систему здоровья героя (3 жизни) и режим неуязвимости после столкновения.

• Реализовать механизм открытия ворот при достижении определенного счета.

• Обеспечить корректное завершение игры при победе (сбор всех печенек) или поражении (потеря всех жизней).

### 4. Компоненты проекта.

• Движение героя с буферизацией направления (ввод сохраняется заранее)

• Столкновения героя со стенами (невозможно пройти сквозь)

• Сквозной проход через левый и правый край экрана (телепортация)

• Система сбора печенек с увеличением счета

• Три жизни героя (отображаются как HP)

• Режим неуязвимости после получения урона (3 секунды)

• 4 призрака разных цветов, движущихся по лабиринту

• Искусственный интеллект призраков на базе алгоритма A* (поиск кратчайшего пути)

• Генерация случайных маршрутов для призраков

• Механизм открытия ворот при 50 очках

• Отображение текущего счета и здоровья на экране

• Экран победы при сборе всех 318 печенек

• Экран поражения при потере всех жизней

• Лабиринт размером 28×31 клетка (28 столбцов, 31 строка)

• Преобразование ASCII-карты в игровой лабиринт (символ X – стена, G – место спавна призраков, пробел – проходимая зона)

### 5. Инструкция по использованию.

• Перед использованием игры нужно убедиться, что установлены все необходимые библиотеки: pygame, numpy, tcod.

• После запуска игры открывается окно с лабиринтом. Желтый герой находится в левом верхнем углу, призраки – внизу карты.

• Управление осуществляется с помощью клавиш-стрелок: **ВВЕРХ, ВНИЗ, ВЛЕВО, ВПРАВО**. Герой движется в выбранном направлении, пока не упрется в стену.

• Задача игрока – собрать все желтые точки (печеньки), разбросанные по лабиринту. Каждая печенька приносит 1 очко.

• При столкновении с призраком игрок теряет одну единицу здоровья (HP) и становится на короткое время неуязвимым (призраки проходят сквозь него без урона).

• Когда игрок набирает 50 очков, ворота в центральной части лабиринта открываются, допуская к ранее закрытой зоне.

• Для полной победы необходимо собрать все 318 печенек в лабиринте. После этого появится экран "YOU WIN!".

• Если здоровье падает до 0, игра завершается экраном "GAME OVER".

• Важное примечание: герой может проходить сквозь левую и правую границы экрана, появляясь с противоположной стороны. Сквозь верхнюю и нижнюю границы проход невозможен.

## Часть 2. Технический документ.

### 1. Цели разработки.

• Создать полноценную пародийную версию Pacman.

• Реализовать систему поиска пути для призраков с использованием библиотеки tcod.

• Отработать навыки объектно-ориентированного программирования: иерархия классов **GameObject → MovableObject → Hero / Ghost.**

• Разработать гибкую систему загрузки уровня из ASCII-схемы с автоматическим распознаванием стен, печенек и точек спавна призраков.

• Реализовать систему коллизий на основе прямоугольников (pygame.Rect) и корректную обработку столкновений с несколькими объектами одновременно.

• Обеспечить плавное управление с буферизацией ввода для более отзывчивой системы игры.

### 2. Использованные технологии.

• **Языки программирования:** Python.

• **Библиотеки:** pygame (графика, ввод, отрисовка), numpy (работа с массивами для поиска пути), tcod (алгоритм поиска пути A*), random (случайные маршруты), enum (направления движения), sys (системные операции).

### 3. Архитектура проекта.

• Проект использует объектно-ориентированную архитектуру с четким разделением ответственности.

• **GameRenderer** – главный контроллер игры: управляет игровым циклом, отрисовкой, обработкой событий, хранением всех игровых объектов (стены, печеньки, герой, призраки), счетом и здоровьем.

• **GameObject** – базовый класс всех объектов на экране: содержит позицию, размер, цвет, форму (круг/прямоугольник), метод draw() для отрисовки.

• **MovableObject** – наследуется от GameObject, добавляет логику движения: буфер направления, проверку коллизий со стенами, автоматическое перемещение.

• **Hero** – наследуется от MovableObject, реализует специфичную логику героя: сбор печенек, столкновения с призраками, режим неуязвимости, телепортацию через края экрана, отрисовку с открытым ртом в зависимости от направления.

• **Ghost** – наследуется от MovableObject, реализует логику призрака: движение по заранее вычисленному пути с помощью A*, запрос нового случайного маршрута при достижении цели.

• **Pathfinder** – класс-обертка над tcod.path.AStar, предоставляет метод get_path() для поиска кратчайшего пути между двумя точками в лабиринте.

• **PacmanGameController** – загрузчик уровня: хранит ASCII-схему лабиринта, преобразует ее в numpy-массив (0 – стена, 1 – проход), создает списки позиций печенек, достижимых клеток и точек спавна призраков.

• **Cookie** – наследуется от GameObject, представляет собой маленькую желтую точку (круг) для сбора.

• **Wall** – наследуется от GameObject, представляет собой стену (синий прямоугольник) с которой нельзя сталкиваться.

• **Direction** – перечисление (Enum) возможных направлений движения: **LEFT, UP, RIGHT, DOWN, NONE.**

### 4. Структура проекта.

• Проект состоит из одного основного файла: Void_Run.py.

• В этом файле последовательно определены:

• Импорты (pygame, numpy, tcod, random, enum)

• Класс Direction (Enum)

• Вспомогательные функции translate_screen_to_maze() и translate_maze_to_screen()

• Базовый класс GameObject

• Класс Wall (наследует GameObject)

• Класс GameRenderer (главный контроллер)

• Класс MovableObject (наследует GameObject)

• Класс Hero (наследует MovableObject)

• Класс Ghost (наследует MovableObject)

• Класс Cookie (наследует GameObject)

• Класс Pathfinder

• Класс PacmanGameController (загрузчик уровня)

• Точка входа (if name == "main")

### 5. Ключевые компоненты системы.

• **GameRenderer** – центральный компонент, управляющий всем. Хранит списки _game_objects, _walls, _cookies, ссылку на _hero. Содержит главный игровой цикл tick(in_fps), который работает, пока _done == False. В каждом кадре вызывает tick() и draw() для всех объектов, обновляет счет и HP, проверяет условия открытия ворот и победы.

• **Система коллизий** – каждый объект имеет метод get_shape(), возвращающий pygame.Rect. Коллизия проверяется через colliderect(). Герой проверяет столкновения со стенами перед каждым движением и с печеньками/призраками после движения.

• **Буферизация ввода** – в Hero есть current_direction (текущее направление) и direction_buffer (запрошенное направление). Если в буфере направление, не ведущее в стену, оно становится текущим. Это позволяет игроку нажать клавишу заранее, и герой повернет, как только это станет возможно.

• **Поиск пути призраков** – Ghost не имеет собственного "ума". Вместо этого он получает от PacmanGameController список точек маршрута (координаты экрана) и движется к ним, пересчитывая направление на каждом шаге. При достижении цели или если следующая цель недостижима, призрак запрашивает новый случайный маршрут.

• **A*** **алгоритм** – реализован через tcod.path.AStar. На вход подается cost-массив (bool: True – проходимо, False – стена). Метод get_path(from_x, from_y, to_x, to_y) возвращает список координат кратчайшего пути.

• **Загрузка уровня** – ASCII-схема размером 31 строка × 28 символов. Символ "X" – стена, "G" – место спавна призрака, пробел – проходимая зона с печенькой, "P" – стартовая позиция героя. Автоматически создаются все стены, печеньки, запоминаются точки спавна призраков.

• **Открытие ворот** – в GameRenderer в начале игры сохраняются определенные стены (y==12, x==13 и x==14) в список _gate_walls. При достижении счета >= 50 эти стены удаляются из _game_objects и _walls, флаг _gate_opened становится True.

• **Система здоровья** – _hp = 3. При столкновении с призраком (если не invincible) вызывается damage_player(), уменьшающий HP. При _hp <= 0 – show_game_over(). После получения урона включается invincible = True на 180 кадров (примерно 3 секунды при 60 FPS, или 1.5 секунды при 120 FPS).

• **Телепортация** – в методе tick() Hero проверяет: если x < 0, то x = ширина экрана; если x > ширина экрана, то x = 0. Это создает эффект тоннеля с левого края на правый.

### 6. Реализация пользовательского интерфейса.

• Интерфейс построен исключительно на Pygame без использования визуальных редакторов.

• Игровое окно имеет размер, вычисляемый динамически: ширина = количество столбцов лабиринта (28) × размер клетки (20) = 560 пикселей; высота = количество строк (31) × 20 = 620 пикселей.

• Лабиринт отрисовывается через стены (синие прямоугольники), фон черный.

• Печеньки – маленькие желтые круги диаметром 4 пикселя, отцентрированные в клетке.

• Герой – желтый круг с вырезанным ртом в виде треугольника. Рот поворачивается в зависимости от текущего направления движения.

• Призраки – квадраты (или круги, в зависимости от is_circle) разных цветов: розово-фиолетовый, красный, голубой, оранжевый.

• Текст счета отображается в верхней части экрана: "Score: X / 318" белым цветом, шрифт Arial 24.

• Текст здоровья отображается слева внизу: "HP: X" белым цветом, шрифт Arial 24.

• Экран победы – зеленая надпись "YOU WIN!" крупным шрифтом (72) по центру экрана, отображается 3 секунды.

• Экран поражения – красная надпись "GAME OVER" крупным шрифтом (72) по центру экрана, отображается 3 секунды.

• Затем был разработан класс MovableObject с базовой логикой движения и проверкой коллизий со стенами.

• После этого был написан класс Hero с управлением от клавиатуры, буферизацией ввода и сбором объектов.

• Далее была создана ASCII-схема лабиринта и класс PacmanGameController для ее парсинга.

• Класс Pathfinder был реализован с использованием библиотеки tcod для поиска пути.

• Затем был написан класс Ghost с логикой движения по точкам маршрута и запросом новых путей.

• Система здоровья, открытие ворот и экраны победы/поражения были добавлены в последнюю очередь.

• Весь процесс сопровождался постоянным тестированием коллизий, корректности движения призраков и баланса сложности.

### 8. Основные сложности и способы их решения.

• **(1) Сложность:** призраки не могли находить путь к случайным точкам в лабиринте, так как координаты экрана не совпадали с координатами массива.

**Решение:** созданы две функции перевода координат: translate_screen_to_maze() – преобразует экранные пиксельные координаты в индексы массива; translate_maze_to_screen() – обратное преобразование. Все операции с поиском пути выполняются в "лабиринтных" координатах, а затем переводятся обратно в экранные.

• **(2) Сложность:** герой мог "застревать" в стенах при быстром нажатии клавиш из-за того, что движение применялось после проверки коллизии.

**Решение:** в Hero добавлена переменная last_non_colliding_position, которая сохраняет последнюю корректную позицию. Если после движения герой оказывается в стене, он возвращается на last_non_colliding_position.

• **(3) Сложность:** печеньки могли быть собраны несколько раз, так как удаление из списка _game_objects не синхронизировалось со списком _cookies.

**Решение:** при сборе печеньки проверяется, что объект присутствует в _game_objects, и удаление производится из основного списка. Список _cookies остается неизменным до следующей итерации, но при проверке коллизий герой сверяется только с _cookies, а удаляет из _game_objects.

• **(4) Сложность:** призраки иногда "замирали" на месте, когда следующая цель в пути была недостижима.

**Решение:** в методе Ghost.calculate_direction_to_next_target() добавлена проверка: если next_target is None, призрак немедленно запрашивает новый случайный путь. Также если разница по X и Y одновременно не равна нулю (движение по диагонали невозможно), запрашивается новый путь.

### 9. Текущие ограничения проекта.

• Весь код находится в одном файле, что усложняет навигацию и поддержку.

• Отсутствуют некоторые механики оригинального Pacman.

• Поведение призраков полностью случайное, нет разных режимов (преследование, разбегание, патрулирование).

• Отсутствует звуковое сопровождение (фоновой музыки и звуковых эффектов).

• Игра не сохраняет лучший результат между запусками.

• При высокой частоте кадров (120 FPS) некоторые временные интервалы (неуязвимость) становятся короче, чем задумано.

• Интерфейс только на английском языке, смена языка не предусмотрена.
